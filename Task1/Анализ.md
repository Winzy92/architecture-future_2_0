# Основные проблемные места текущего решения

| №  | Проблема                               | Описание                                                                                                                                                                                                                                                                                                                                                                                                      |
| -- |----------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1  | Монолитный DWH с бизнес-логикой        | DWH превратился в помойку с кучей зависимостей. Все сервисы связаны с ним. Любая интеграция или реализация новых фич сопровожждается рисками сломать что то существующее. Сейчас он блокирует развитие доменов. Нет контроля данных, все лежит в куче, и там же бизнес логика.                                                                                                                                |
| 2  | Сложные ETL                            | Отчёты строятся часами из-за сложных зависимостей и сложных запросов.                                                                                                                                                                                                                                                                                                                                         |
| 3  | ESB единая точка взаимодействия        | ESB мини монолит, выполяет следующие функции: роутинг, оркестрация, транфсорфмация, в нем есть бизнес логика. Необходимо декомпозировать его, транспорт отдаем на сторону kafka, бизнес логика уходит в домены, контроль данных в data catalog, трансформации в batch processing. Т.о. мы распределяем зоны отвественности, устраняя узкое место, каждая зона отвественности может масштабироваться отдельно. |
| 4  | Нет доменной ответственности за данные | Данные всех доменов хранятся в едином месте.                                                                                                                                                                                                                                                                                                                                                                  |
| 5  | BI зависит от DWH                      | Низкая производительность, запросы без промежуточного слоя обрабываются в DWH.                                                                                                                                                                                                                                                                                                                                |
| 6  | Легаси SQL Server 2008 и PowerBuilder  | Устаревшие технологии дороже в поддержке, особенно если продукт развивается.                                                                                                                                                                                                                                                                                                                                  |
| 7  | Отсутствие self-service аналитики      | Аналитика сейчас зависит от специалистов IT, нет опции самим построить отчет из-за сложного DWH.                                                                                                                                                                                                                                                                                                              |
| 8  | Нет разделения чувствительных данных   | Риск утечки чувствительных данных                                                                                                                                                                                                                                                                                                                                                                             |
| 9  | Нет real-time обработки                | Невозможность быстрых решений                                                                                                                                                                                                                                                                                                                                                                                 |

# Приоритизация проблем

## MoSCoW

### Критично
1. Монолитный DWH, блокирует масштабирование.
2. Медленные отчёты. Медленный процесс принятия решений, упущенная прибыль.
3. Нет доменной модели, мы не можем развивать различные направления бизнеса (например фин.тех.).
4. Нет слоя отвечающего за управления данными (доступ, разделение данных по категориям и тп.). Высокие риски утечки данных, судебные издержки, санкции регулятора.

### Важно
1. ESB как bottleneck.
2. BI зависит от DWH, низкая производительность. SQL выполняется напрямую, из-за сложности запроса может выполняться очень долго.
3. Нет self-service

### Желательно
1. Отсутствие real-time, важный пункт для фин. теха и AI, а также SLA. Но прямо сейчас это не критично.
2. Легаси UI (PowerBuilder), устарел, но прямо сейчас не является критичным.


## Приоритизация через бизнес-ценность

### Выручка:
1. Фин.тех. аналитика, скорость принятия решений на ее основе.
2. Time to market новых продуктов.

### Скорость работы:
1. Отказ от DWH как ядра
2. Разделение данных на слои (Data Products)

### Риски:
1. Внедрения слоя контроля данных и доступа к данным, разделение чувсивтельных данных и соблюдение ФЗ.


# Предлогаемые решения:

1. Внедряем Data-Mesh, каждый домен владеет своим набором данных.
2. Lakehouse вместо DWH, получим масштабируемость и производительность.
3. Kafka вместо ESB. Хорошо масштабируется, просто интегрировать новые сервисы.
4. Разарабатываем слой контроля данных (доступ к данным, происхождение, классификация и тп).
5. Разрабатываем единую витрину данных для отчетов (Self-Service BI).

# Этапы перезда с DWH

Предлагаю использовать паттерн Strangler Fig, будем переносить по частям.
1. Замораживаем DWH, не пишем в него данные.
2. Разворачиваем lakehouse, data platform.
3. DWH выгружаем в lakehouse
4. Реализуем новые data products, данные идут не из DWH.
5. BI миграция, переносим данные из DWH в data products.
6. Начинаем по частям отключать DWH.
